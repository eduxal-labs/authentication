Authentication API Reference

Passwordless, phone-based OTP authentication for the Eduxal platform. Identities are verified via a 6-digit code delivered over WhatsApp. Short-lived and long-lived tokens are used to manage authenticated access.

Base URL https://auth.eduxal.com
OTP Delivery WhatsApp
Tokens Bearer tokens

Authentication Flows

New User Registration

1
POST /auth/login — Send { phone }
→ OTP sent to phone via WhatsApp. Returns { phone, created, ttl }.
2
POST /auth/verify — Send { phone, code, device }
→ Phone verified; no existing account detected. Returns { token } — a short-lived setup token.
3
POST /auth/setup — Header: Setup token. Body: { name, device }
→ Account created. Returns { session, access_token, refresh_token, user }.

Returning User Login

1
POST /auth/login — Send { phone }
2
POST /auth/verify — Send { phone, code, device }
→ Existing account found. Returns { session, access_token, refresh_token, user }.

Token Refresh

1
GET /auth/refresh — Header: Refresh token
→ A fresh token pair is issued for the current session.

Token System

The API uses bearer tokens for authenticated access. Present tokens in the Authorization: Bearer <token> header. Different token types are used for access, refresh, and account setup flows.

Access Token
Purpose: Authenticated API access
TTL: 2 days
Subject: Authenticated user
Used on all authenticated endpoints.
Refresh Token
Purpose: Session renewal
TTL: 52 days
Subject: Authenticated user
Only valid on GET /auth/refresh.
Setup Token
Purpose: Account setup
TTL: 15 minutes
Subject: Verified phone number
Only valid on POST /auth/setup.

Token Extraction Errors

These errors apply to endpoints that require a token and may be returned before endpoint-specific validation runs:

Condition Status Message
Authorization header missing or not UTF-8 401 unauthorized
Header present but not prefixed with Bearer 401 unauthorized
Token is invalid or malformed 400 invalid token
Token has expired 401 unauthorized
Token is the wrong type (e.g., refresh where access expected) 403 permission denied

Authentication Endpoints

POST /auth/login Send OTP to phone

Sends a 6-digit OTP to the given phone number using WhatsApp. This is the first step of the authentication flow. The OTP is never returned in the response — it arrives on WhatsApp only.

🔓 No authentication required

Request Body

JSON 
{
  "phone": "+254759762268"   // required — E.164 format (starts with +)
}
Field Type Validation
phone string required Must start with + and be a valid E.164 phone number.

Success Response — 200 OK

JSON 
{
  "phone":   "+254759762268",
  "created": "2025-04-26T07:30:00Z",   // when the OTP was generated
  "ttl":     "2025-04-26T07:45:00Z"    // OTP expiry time
}
⚠️ The OTP code is never present in the response. It is delivered exclusively over WhatsApp.

Error Responses

Status Message Cause
400 Request validation error Request body is invalid JSON, or Content-Type is missing or not application/json.
422 Request validation error phone field is missing or fails E.164 validation.
429 too many requests, Please slow down An OTP was already sent to this phone number within the past 60 seconds.
500 internal server error A downstream delivery or storage error occurred.

Notes

  • Repeated requests for the same phone number are limited for 60 seconds between successful OTP sends.
  • A new successful call invalidates any previously issued OTP for that phone number.
  • The phone number must be able to receive WhatsApp messages.
POST /auth/verify Validate OTP

Validates the OTP received via WhatsApp. The response depends on whether an account already exists for the phone number. For new users, returns a short-lived setup token. For existing users, returns authenticated session details.

🔓 No authentication required

Request Body

JSON 
{
  "phone":   "+254759762268",   // required — E.164
  "code":    "136483",         // required — 6-digit OTP from WhatsApp
  "device":  "Ubuntu Linux, Bruno"  // required — device label
}
Field Type Description
phone string required E.164 phone number.
code string required 6-digit OTP received on WhatsApp.
session string | null optional Optional session identifier. If accepted, the existing session may be reused.
device string required Human-readable device label. Ignored for new users (no session is created).

Success Response — 200 OK (Existing User)

JSON 
{
  "session":       "69ed056664bc5d80849a9322",
  "access_token":  "<access-token>",   // valid 2 days
  "refresh_token": "<refresh-token>",  // valid 52 days
  "user": {
    "id":       "507f1f77bcf86cd799439011",
    "phone":    "+254759762268",
    "name":     "Abdihakim Osman",
    "level":    "Normal",   // "Normal" | "System" | "Super"
    "status":   "Active",   // "Active" | "Invited" | "Blocked" | "Deleted"
    "profiled": true,
    "created":  "2025-01-15T10:30:00Z"
  }
}

Success Response — 200 OK (New / Unregistered User)

JSON 
{
  "token": "<setup-token>"   // Setup token — valid 15 minutes only
}
ℹ️ The response does not include a dedicated type field. Detect the variant by shape: token key → new user; session + access_token keys → existing user.

Error Responses

Status Message Cause
400 invalid verification code Submitted OTP is incorrect.
400 invalid phone number... phone fails E.164 validation.
403 permission denied The account is not allowed to complete this action.
404 verification code not found No active verification request was found for this phone number.
500 internal server error A server-side error occurred.

Session Reuse

On success, the response includes the session identifier that should be used by the client for any subsequent session-related actions. Always read session from the response.

Notes

  • New users receive a short-lived setup token to complete registration.
  • Existing users receive authenticated session details and tokens.
POST /auth/setup Complete account registration

Completes account registration for a newly verified phone number. Requires the setup token returned by POST /auth/verify for new users. On success, creates the user account and opens a new session.

🔑 Requires setup token

Request

HTTP 
POST /auth/setup
Authorization: Bearer <setup-token>
Content-Type:  application/json

{
  "name":   "Abdihakim Osman",   // required — display name
  "device": "Ubuntu Linux, Bruno"   // required — device label
}

Success Response — 200 OK

JSON 
{
  "session":       "69ed056664bc5d80849a9322",
  "access_token":  "<access-token>",
  "refresh_token": "<refresh-token>",
  "user": {
    "id":       "507f1f77bcf86cd799439011",
    "phone":    "+254759762268",
    "name":     "Abdihakim Osman",
    "level":    "Normal",
    "status":   "Active",
    "profiled": false,   // always false for new accounts
    "created":  "2025-04-26T07:55:00Z"
  }
}

Error Responses

Status Message Cause
400 invalid token The provided token is invalid or malformed.
401 unauthorized Header missing, wrong prefix, or Setup token is expired (> 15 min).
403 permission denied The provided token cannot be used for this action, or the account is not allowed to proceed.
422 Request validation error name or device missing from body.
500 internal server error An unexpected server-side error occurred.

Notes

  • The verified phone number is derived from the setup token, not from the request body.
  • New users receive a default account profile on successful setup.
GET /auth/refresh Rotate token pair

Exchanges a valid refresh token for a fresh access and refresh token pair. No request body is needed.

🔑 Requires refresh token

Request

HTTP 
GET /auth/refresh
Authorization: Bearer <refresh-token>

Success Response — 200 OK

Returns the same response shape as POST /auth/setup, including a fresh access_token, refresh_token, session identifier, and user object.

Error Responses

Status Message Cause
400 invalid token The provided token is invalid or malformed.
401 unauthorized Header missing, wrong prefix, or Refresh token is expired (> 52 days).
403 permission denied The token cannot be used for this action, or the account/session is no longer eligible for refresh.
404 user not found The associated user account could not be found.
500 internal server error A server-side error occurred.

Notes

  • If refresh fails because the token is expired or invalid, the user should sign in again.
  • Always replace stored tokens with the latest values returned by this endpoint.

User Endpoints

GET /user Get current user

Returns the full profile of the currently authenticated user.

🔑 Requires access token

Request

HTTP 
GET /user
Authorization: Bearer <access-token>

Success Response — 200 OK

JSON 
{
  "id":       "507f1f77bcf86cd799439011",
  "phone":    "+254759762268",
  "name":     "Abdihakim Osman",
  "level":    "Normal",
  "status":   "Active",
  "profiled": true,
  "created":  "2025-01-15T10:30:00Z"
}

Error Responses

Status Message Cause
401 unauthorized Header missing, wrong prefix, or token expired.
403 permission denied Token is a Refresh or Setup token.
404 user not found User record does not exist (account deleted after token issuance).
500 internal server error An unexpected server-side error occurred.
⚠️ Always inspect the returned status field in your application logic before enabling privileged user actions.
PATCH /user/rename Update display name

Updates the authenticated user's display name and returns the full updated user record. No need to call GET /user afterwards.

🔑 Requires access token

Request

HTTP 
PATCH /user/rename
Authorization: Bearer <access-token>
Content-Type:  application/json

{
  "name": "Alagha"   // required — new display name
}

Success Response — 200 OK

Full updated User object (same shape as GET /user) with name reflecting the new value.

Error Responses

Status Message Cause
401 unauthorized Header missing, wrong prefix, or token expired.
403 permission denied Token is a Refresh or Setup token.
415 Request validation error Content-Type is not application/json.
422 Request validation error name field is missing or not a JSON string.
500 internal server error An unexpected server-side error occurred.

Notes

  • Clients should validate display names before submission.
POST /user/change-phone Initiate phone change

Initiates a phone number change by sending a 6-digit OTP to the new phone number via WhatsApp. The change is not committed until confirmed via PATCH /user/confirm-change-phone.

🔑 Requires access token

Request

HTTP 
POST /user/change-phone
Authorization: Bearer <access-token>
Content-Type:  application/json

{
  "phone": "+254759766444"   // required — the new phone number (E.164)
}

Success Response — 200 OK

Same shape as POST /auth/login. The OTP code is not in the response.

Error Responses

Status Message Cause
400 nothing new to update Submitted phone is the same as the user's current phone.
400 invalid phone number... Phone fails E.164 validation.
401 unauthorized Header missing, wrong prefix, or token expired.
403 permission denied Token is wrong type.
404 user not found User record does not exist.
409 record already exists Another user already uses this phone number.
429 too many requests, Please slow down OTP already sent to this target phone within the past 60 seconds.
500 internal server error An unexpected server-side error occurred.

Notes

  • Repeated requests for the same target phone number may be temporarily limited.
  • If the target phone number is already assigned to another user, the endpoint returns 409.
  • Complete confirmation promptly after receiving the OTP.
PATCH /user/confirm-change-phone Confirm phone change

Validates the OTP sent to the new phone number and commits the change to the user's record.

🔑 Requires access token

Request

HTTP 
PATCH /user/confirm-change-phone
Authorization: Bearer <access-token>
Content-Type:  application/json

{
  "phone": "+254759766444",   // required — the new phone number
  "code":  "861877"            // required — 6-digit OTP
}

Success Response — 200 OK

Full updated User object with phone reflecting the new number.

Error Responses

Status Message Cause
400 invalid verification code OTP does not match the stored code.
401 unauthorized Header missing, wrong prefix, or token expired.
403 permission denied Token is wrong type.
404 verification code not found No active verification request was found for this phone number.
404 user not found User record deleted between calls.
409 record already exists Another user already uses this phone number.
500 internal server error An unexpected server-side error occurred.

Notes

  • The authenticated user is derived from the access token.
  • Use the same phone number that was submitted to POST /user/change-phone.

Session Endpoints

GET /sessions List all sessions

Returns all sessions belonging to the authenticated user. An empty array is a valid, non-error response.

🔑 Requires access token

Request

HTTP 
GET /sessions
Authorization: Bearer <access-token>

Success Response — 200 OK

JSON 
[
  {
    "id":      "69ed056664bc5d80849a9322",
    "device":  "Ubuntu Linux, Bruno",
    "created": "2025-04-01T10:00:00Z",
    "ttl":     "2025-07-10T10:00:00Z"
  }
]

Error Responses

Status Message Cause
401 unauthorized Header missing, wrong prefix, or token expired.
403 permission denied Token is a Refresh or Setup token.
500 internal server error An unexpected server-side error occurred.

Notes

  • The response is a top-level JSON array, not { "sessions": [...] }.
  • Use the returned id value when revoking a session.
DELETE /sessions/{id} Revoke a session

Deletes a specific session, immediately preventing any future token refresh for it. This is the "log out of a device" mechanism. The operation is idempotent — deleting a non-existent session returns 200.

🔑 Requires access token

Request

HTTP 
DELETE /sessions/69ed056664bc5d80849a9322
Authorization: Bearer <access-token>
Path Param Type Description
id string Session identifier to delete. Use the id field returned by GET /sessions.

Success Response — 200 OK

JSON 
null   // returned whether the session was deleted or did not exist

Error Responses

Status Message Cause
400 invalid id Path segment {id} is not a valid session identifier.
401 unauthorized Header missing, wrong prefix, or token expired.
403 permission denied Token is wrong type; or session exists but belongs to a different user.
500 internal server error An unexpected server-side error occurred.
🚨 Deleting your own active session: After revoking the current session, clear any locally stored credentials and require the user to sign in again if needed.

Notes

  • Deleting a missing session is treated as a successful no-op.
  • Use the session id returned by GET /sessions.

Success Response — 200 OK

JSON 
{ "status": "Ok" }
