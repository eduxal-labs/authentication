Authentication API Reference

Passwordless, phone-based OTP authentication for the Eduxal platform. Identities are verified via a 6-digit code delivered over WhatsApp. All tokens are PASETO v4-local (symmetric encryption).

Base URL https://auth.eduxal.com
Runtime AWS Lambda · arm64
Database DynamoDB
OTP Delivery WhatsApp Business API
Tokens PASETO v4-local

Authentication Flows

New User Registration

1
POST /auth/login — Send { phone }
→ OTP sent to phone via WhatsApp. Returns { phone, created, ttl }.
2
POST /auth/verify — Send { phone, code, session: null, device }
→ Phone verified; no existing account detected. Returns { token } — a 15-min Setup token.
3
POST /auth/setup — Header: Setup token. Body: { name, device }
→ Account created. Returns { session, access_token, refresh_token, user }.

Returning User Login

1
POST /auth/login — Send { phone }
2
POST /auth/verify — Send { phone, code, session, device }
→ Existing account found. Returns { session, access_token, refresh_token, user }.

Token Refresh

1
GET /auth/refresh — Header: Refresh token
→ New access + refresh token pair issued. Session ID unchanged.

Token System

All tokens are PASETO v4-local (symmetric encryption). Each token's payload contains the user's subject, session ID, purpose discriminant, and expiry timestamp. Present tokens in the Authorization: Bearer <token> header.

Access Token
Kind: 1
TTL: 2 days
Subject: User ID
Used on all authenticated endpoints.
Refresh Token
Kind: 2
TTL: 52 days
Subject: User ID
Only valid on GET /auth/refresh.
Setup Token
Kind: 3
TTL: 15 minutes
Subject: Phone number
Only valid on POST /auth/setup.

Token Extraction Errors

These errors apply to every endpoint that requires a token, evaluated before handler logic runs:

Condition Status Message
Authorization header missing or not UTF-8 401 unauthorized
Header present but not prefixed with Bearer 401 unauthorized
Token cannot be decrypted / malformed 400 invalid token
Token has expired 401 unauthorized
Token is the wrong type (e.g., refresh where access expected) 403 permission denied

Authentication Endpoints

POST /auth/login Send OTP to phone

Sends a 6-digit OTP to the given phone number via the WhatsApp Business API. Always the first step of any authentication or login flow. The OTP is never returned in the response — it arrives on WhatsApp only.

🔓 No authentication required

Request Body

JSON 
{
  "phone": "+254759762268"   // required — E.164 format (starts with +)
}
Field Type Validation
phone string required Must start with + and pass E.164 validation. Validated at deserialization.

Success Response — 200 OK

JSON 
{
  "phone":   "+254759762268",
  "created": "2025-04-26T07:30:00Z",   // when the OTP was generated
  "ttl":     "2025-04-26T07:45:00Z"    // created + 15 minutes
}
⚠️ The OTP code field is annotated #[serde(skip)] and is never present in the response. It is delivered exclusively over WhatsApp.

Error Responses

Status Message Cause
400 Axum plain-text rejection Body is not valid JSON, or Content-Type is missing/not application/json.
422 Axum plain-text rejection phone field is missing or fails E.164 validation. Note: response body is plain text, not JSON.
429 too many requests, Please slow down An OTP was already sent to this phone number within the past 60 seconds.
500 internal server error WhatsApp API delivery failed, or DynamoDB read/write failed.

Notes

  • Rate limiting is keyed on the phone number, not client IP. Any client can trigger the limit for a given number.
  • A new successful call overwrites any previous OTP — prior codes are immediately invalidated.
  • OTP delivery (WhatsApp send) happens before the record is persisted to DynamoDB. A failed DynamoDB write means the user received a code that cannot be verified.
  • The phone number must be registered on WhatsApp. There is no SMS fallback.
POST /auth/verify Validate OTP

Validates the OTP received via WhatsApp. The response shape depends on whether an account exists for the phone number. For new users, returns a short-lived Setup token. For existing users, returns a full token pair and session.

🔓 No authentication required

Request Body

JSON 
{
  "phone":   "+254759762268",   // required — E.164
  "code":    "136483",         // required — 6-digit OTP from WhatsApp
  "session": null,            // optional — 24-char hex ObjectId to reuse
  "device":  "Ubuntu Linux, Bruno"  // required — stored on the session
}
Field Type Description
phone string required E.164 phone number.
code string required 6-digit OTP received on WhatsApp.
session string | null optional 24-char hex ObjectId. If provided and owned by this user, the session is reused.
device string required Human-readable device label. Ignored for new users (no session is created).

Success Response — 200 OK (Existing User)

JSON 
{
  "session":       "69ed056664bc5d80849a9322",
  "access_token":  "v4.local.ukuGtEfqElk6...",   // valid 2 days
  "refresh_token": "v4.local.rPWuvV7iMHrS...",  // valid 52 days
  "user": {
    "id":       "507f1f77bcf86cd799439011",
    "phone":    "+254759762268",
    "name":     "Abdihakim Osman",
    "level":    "Normal",   // "Normal" | "System" | "Super"
    "status":   "Active",   // "Active" | "Invited" | "Blocked" | "Deleted"
    "profiled": true,
    "created":  "2025-01-15T10:30:00Z"
  }
}

Success Response — 200 OK (New / Unregistered User)

JSON 
{
  "token": "v4.local.zXZ3HE1F7aR5..."   // Setup token — valid 15 minutes only
}
ℹ️ The response is untagged — there is no type discriminant field. Detect the variant by shape: token key → new user; session + access_token keys → existing user.

Error Responses

Status Message Cause
400 invalid verification code Submitted OTP does not match the stored code.
400 invalid phone number... phone fails E.164 validation.
400 invalid id session is not a valid 24-hex ObjectId.
403 permission denied User account is Blocked.
404 verification code not found POST /auth/login was not called first, or the record was evicted from DynamoDB.
500 internal server error Any DynamoDB operation failed.

Session Reuse

When session is provided, the server fetches it and checks ownership. If the session exists and belongs to the authenticated user, its ID is reused (TTL and refresh token are reset). Otherwise, a new session is generated. Always read session from the response.

Notes

  • Blocked users are checked twice — before and after OTP validation — to close a race window.
  • The verification record is not deleted after success. The same code can be replayed until DynamoDB TTL evicts it (up to 48 hours after expiry).
  • For new users, device and session are silently discarded — no session record is created.
POST /auth/setup Complete account registration

Completes account registration for a newly verified phone number. Requires the Setup token returned by POST /auth/verify when the phone was unrecognized. On success, creates the user record and opens a new session.

🔑 Requires Setup token (KIND = 3, TTL = 15 min)

Request

HTTP 
POST /auth/setup
Authorization: Bearer v4.local.zXZ3HE1F7aR5...
Content-Type:  application/json

{
  "name":   "Abdihakim Osman",   // required — display name
  "device": "Ubuntu Linux, Bruno"   // required — device label
}

Success Response — 200 OK

JSON 
{
  "session":       "69ed056664bc5d80849a9322",
  "access_token":  "v4.local.ukuGtEfqElk6...",
  "refresh_token": "v4.local.rPWuvV7iMHrS...",
  "user": {
    "id":       "507f1f77bcf86cd799439011",
    "phone":    "+254759762268",
    "name":     "Abdihakim Osman",
    "level":    "Normal",
    "status":   "Active",
    "profiled": false,   // always false for new accounts
    "created":  "2025-04-26T07:55:00Z"
  }
}

Error Responses

Status Message Cause
400 invalid token Bearer token cannot be decrypted or is malformed.
401 unauthorized Header missing, wrong prefix, or Setup token is expired (> 15 min).
403 permission denied Token is an Access or Refresh token (wrong type). Also: user is Blocked.
422 Axum plain-text rejection name or device missing from body.
500 internal server error DynamoDB read or write failed.

Notes

  • The phone comes from the token, not the body. The body only sets the display name and device label.
  • If the phone already has a user record (e.g., called twice), the user is renamed and a new session is created. Not idempotent for existing users.
  • For blocked users, the rename is committed to DynamoDB before the 403 is returned — a side effect that cannot be rolled back.
  • New users always receive level: "Normal", status: "Active", and profiled: false.
GET /auth/refresh Rotate token pair

Exchanges a valid Refresh token for a fresh Access + Refresh token pair. The session ID does not change. No request body needed — all required data is inside the encrypted token.

🔑 Requires Refresh token (KIND = 2, TTL = 52 days)

Request

HTTP 
GET /auth/refresh
Authorization: Bearer v4.local.rPWuvV7iMHrS...

Success Response — 200 OK

Same shape as POST /auth/setup. Both access_token and refresh_token are freshly generated. session is always the same ID that was in the Refresh token.

Error Responses

Status Message Cause
400 invalid token Token cannot be decrypted or is malformed.
401 unauthorized Header missing, wrong prefix, or Refresh token is expired (> 52 days).
403 permission denied Token is wrong type; or user is Blocked; or the session document no longer exists (was deleted via DELETE /sessions/{id}).
404 user not found User record was deleted from DynamoDB after token issuance.
500 internal server error DynamoDB read or write failed.

Notes

  • 401 = session expired (token TTL exceeded) → user must re-authenticate from login.
  • 403 (missing session) = session was revoked → clear credentials, redirect to login.
  • 404 = account deleted → do not retry.
  • Old refresh tokens technically remain usable until their 52-day PASETO expiry because the endpoint does not validate that the token ID matches session.refresh. Always replace the stored refresh token with the one from the response.

User Endpoints

GET /user Get current user

Returns the full profile of the currently authenticated user.

🔑 Requires Access token (KIND = 1)

Request

HTTP 
GET /user
Authorization: Bearer v4.local.ukuGtEfqElk6...

Success Response — 200 OK

JSON 
{
  "id":       "507f1f77bcf86cd799439011",   // 24-char hex ObjectId
  "phone":    "+254759762268",            // E.164
  "name":     "Abdihakim Osman",
  "level":    "Normal",                // "Normal" | "System" | "Super"
  "status":   "Active",                // "Active" | "Invited" | "Blocked" | "Deleted"
  "profiled": true,
  "created":  "2025-01-15T10:30:00Z"      // RFC 3339 UTC, second precision
}

Error Responses

Status Message Cause
401 unauthorized Header missing, wrong prefix, or token expired.
403 permission denied Token is a Refresh or Setup token.
404 user not found User record does not exist (account deleted after token issuance).
500 internal server error DynamoDB read failed or record is corrupt.
⚠️ A 200 response does not guarantee the user is active. The status field may be "Blocked" or "Deleted". Always inspect the status field in your application logic.
PATCH /user/rename Update display name

Updates the authenticated user's display name and returns the full updated user record. No need to call GET /user afterwards.

🔑 Requires Access token (KIND = 1)

Request

HTTP 
PATCH /user/rename
Authorization: Bearer v4.local.ukuGtEfqElk6...
Content-Type:  application/json

{
  "name": "Alagha"   // required — new display name
}

Success Response — 200 OK

Full updated User object (same shape as GET /user) with name reflecting the new value.

Error Responses

Status Message Cause
401 unauthorized Header missing, wrong prefix, or token expired.
403 permission denied Token is a Refresh or Setup token.
415 Axum plain-text rejection Content-Type is not application/json.
422 Axum plain-text rejection name field is missing or not a JSON string.
500 internal server error DynamoDB update failed.

Notes

  • There is no duplicate-name check — sending the current name writes to DynamoDB and returns 200.
  • No name validation at the API layer — empty strings and arbitrarily long names are accepted. Enforce limits client-side.
  • DynamoDB's UpdateItem is an upsert. If the user record was deleted, a partial ghost record will be created, causing subsequent reads to fail with 500.
POST /user/change-phone Initiate phone change

Initiates a phone number change by sending a 6-digit OTP to the new phone number via WhatsApp. The change is not committed until confirmed via PATCH /user/confirm-change-phone.

🔑 Requires Access token (KIND = 1)

Request

HTTP 
POST /user/change-phone
Authorization: Bearer v4.local.ukuGtEfqElk6...
Content-Type:  application/json

{
  "phone": "+254759766444"   // required — the NEW phone number (E.164)
}

Success Response — 200 OK

Same shape as POST /auth/login. The OTP code is not in the response.

Error Responses

Status Message Cause
400 nothing new to update Submitted phone is the same as the user's current phone.
400 invalid phone number... Phone fails E.164 validation.
401 unauthorized Header missing, wrong prefix, or token expired.
403 permission denied Token is wrong type.
404 user not found User record does not exist.
429 too many requests, Please slow down OTP already sent to this target phone within the past 60 seconds.
500 internal server error WhatsApp delivery failed or DynamoDB write failed.

Notes

  • Rate limiting is keyed on the target phone number, not the user's ID.
  • The ttl in the response is informational — the confirm endpoint does not enforce it at the application level.
  • The new phone number is not reserved during the pending window; another user could register with it.
PATCH /user/confirm-change-phone Confirm phone change

Validates the OTP sent to the new phone number and commits the change to the user's record.

🔑 Requires Access token (KIND = 1)

Request

HTTP 
PATCH /user/confirm-change-phone
Authorization: Bearer v4.local.ukuGtEfqElk6...
Content-Type:  application/json

{
  "phone": "+254759766444",   // required — the NEW phone number (same as change-phone)
  "code":  "861877"            // required — 6-digit OTP as string (leading zeros matter)
}

Success Response — 200 OK

Full updated User object with phone reflecting the new number.

Error Responses

Status Message Cause
400 invalid verification code OTP does not match the stored code.
401 unauthorized Header missing, wrong prefix, or token expired.
403 permission denied Token is wrong type.
404 verification code not found change-phone was not called first, or DynamoDB TTL evicted the record.
404 user not found User record deleted between calls.
500 internal server error DynamoDB read or write failed.

Notes

  • The user identity comes from the token, not the body. Any user with a valid access token and a valid OTP for a phone can confirm a change to that phone.
  • The verification record is not deleted after success — the same code can be replayed.
  • No phone uniqueness enforcement — two users can end up with the same phone number.
  • No brute-force protection — failed code attempts are not rate-limited or counted.
  • The access token must remain valid for the entire two-step flow. Use GET /auth/refresh if needed before calling this endpoint.

Session Endpoints

GET /sessions List all sessions

Returns all sessions belonging to the authenticated user. An empty array is a valid, non-error response.

🔑 Requires Access token (KIND = 1)

Request

HTTP 
GET /sessions
Authorization: Bearer v4.local.ukuGtEfqElk6...

Success Response — 200 OK

JSON 
[
  {
    "id":      "69ed056664bc5d80849a9322",   // use this to delete the session
    "user":    "507f1f77bcf86cd799439011",
    "refresh": "507f1f77bcf86cd799439013",   // ID of last issued refresh token — treat as opaque
    "device":  "Ubuntu Linux, Bruno",
    "created": "2025-04-01T10:00:00Z",
    "ttl":     "2025-07-10T10:00:00Z"       // created + 100 days — informational only
  }
]

Error Responses

Status Message Cause
401 unauthorized Header missing, wrong prefix, or token expired.
403 permission denied Token is a Refresh or Setup token.
500 internal server error DynamoDB query failed.

Notes

  • The response is a top-level JSON array, not { "sessions": [...] }.
  • The session used to authenticate this request is included in the results.
  • Sessions past their ttl may still appear — DynamoDB TTL cleanup is eventually consistent (up to 48 hours delay). Do not use ttl as a real-time expiry signal.
DELETE /sessions/{id} Revoke a session

Deletes a specific session, immediately preventing any future token refresh for it. This is the "log out of a device" mechanism. The operation is idempotent — deleting a non-existent session returns 200.

🔑 Requires Access token (KIND = 1)

Request

HTTP 
DELETE /sessions/69ed056664bc5d80849a9322
Authorization: Bearer v4.local.ukuGtEfqElk6...
Path Param Type Description
id string 24-char hex MongoDB ObjectId of the session to delete (use id field from GET /sessions).

Success Response — 200 OK

JSON 
null   // returned whether the session was deleted or did not exist

Error Responses

Status Message Cause
400 invalid id Path segment {id} is not a valid 24-char hex ObjectId.
401 unauthorized Header missing, wrong prefix, or token expired.
403 permission denied Token is wrong type; or session exists but belongs to a different user.
500 internal server error DynamoDB read or write failed.
🚨 Deleting your own active session: The delete succeeds immediately. Your current access token remains valid until its 2-day PASETO expiry, but all subsequent GET /auth/refresh calls will return 403. Discard both tokens after deleting your own session.

Notes

  • 403 (ownership mismatch) only fires when the session exists and belongs to another user. A missing session ID always returns 200.
  • The {id} in the URL is the session's own id field — not the refresh field also visible in GET /sessions.
  • The operation makes two DynamoDB calls: a GetItem (ownership check) then a DeleteItem.

Health Check

GET /health Service liveness check

Returns a static 200 OK confirming the Lambda function is running. No authentication required.

Success Response — 200 OK

JSON 
{ "status": "Ok" }

Error Reference

All service-layer errors return a JSON body { "message": "..." }. Errors from Axum's extractor layer (JSON parse failures, wrong Content-Type, missing fields) return plain-text bodies and bypass this envelope.

Status Message Variant Trigger
400 verification code not found VerificationCodeNotFound No OTP record in DynamoDB for the given phone (also returned as 404 in some paths).
400 invalid verification code InvalidVerificationCode OTP code does not match the stored value.
400 invalid phone number. Make sure to include country code InvalidPhoneNumber Phone fails E.164 validation (no + prefix, or invalid structure).
400 invalid id InvalidId A path or body ID is not a valid 24-char hex MongoDB ObjectId.
400 invalid token InvalidToken PASETO token cannot be decrypted or fails JSON parsing.
400 nothing new to update UptoDate change-phone received the user's current phone number.
401 unauthorized Unauthorized Missing Authorization header, wrong prefix, or expired token.
403 permission denied Forbidden Wrong token type; blocked user; missing session (on refresh); or cross-user session delete attempt.
404 verification code not found VerificationCodeNotFound No OTP record exists for the phone — login not called first, or record evicted by DynamoDB TTL.
404 user not found UserNotFound User record does not exist in DynamoDB.
404 invalid session InvalidSession Session ID does not resolve to a session record.
409 record already exists RecordAlreadyExists A conflicting record already exists in the database.
429 too many requests, Please slow down SlowDown OTP requested within the 60-second rate-limit window for a phone number.
500 internal server error InternalServerError Database error, WhatsApp API error, or data corruption. Details are logged server-side only.
